== Architecture Decisions

Each decision is recorded as a lightweight Architecture Decision Record (ADR).

=== ADR-1 — Full-Stack Rust with Leptos and Axum

[cols="1,3", options="header"]
|===
|Attribute |Detail

|Status
|Accepted

|Context
|The project is a proof of concept for a full-stack web application written
entirely in Rust.  A frontend technology must be chosen that satisfies both
the SPA requirement (instant feedback, no full page reloads) and the Rust-only
constraint.

|Decision
|Use Leptos as the SPA framework (compiled to WebAssembly) with Axum as the
HTTP transport layer.  Leptos server functions provide the data layer; no
separate REST API is designed.

|Alternatives Considered
a|
* *Yew* — React-like WASM framework.  More boilerplate than Leptos; weaker
  server-function story.  Rejected.
* *HTMX + Axum* — Server-rendered partials with HTMX swap.  Not a true SPA;
  would require JavaScript for autocomplete and reactive search.  Rejected.
* *JavaScript SPA (React/Svelte) + Axum REST API* — Violates the Rust-only
  constraint.  Rejected.

|Consequences
a|
* Positive: End-to-end type safety.  No manually written API contract.
  Single Cargo workspace for all code.
* Positive: Leptos server functions eliminate HTTP boilerplate between
  frontend and backend.
* Negative: Longer compile times than a JS frontend.
* Negative: Leptos ecosystem is less mature than React; fewer community
  resources.
* Negative: WASM bundle size is larger than a comparable JS bundle.
|===

=== ADR-2 — SQLite with FTS5 as the Only Data Store

[cols="1,3", options="header"]
|===
|Attribute |Detail

|Status
|Accepted

|Context
|A persistence mechanism is needed that supports structured storage and
full-text search with zero external services.

|Decision
|Use SQLite with the FTS5 extension as the sole data store.
The database is a single file on a named Docker volume.

|Alternatives Considered
a|
* *PostgreSQL* — Full-featured RDBMS with excellent full-text search.
  Requires a separate container and connection management.  Overkill for a
  single-user personal tool.  Rejected.
* *Meilisearch / Typesense* — Dedicated search engines.  Additional container,
  additional operational complexity.  Rejected.
* *Plain SQLite without FTS5* — Would require `LIKE`-based search, which is
  slow and cannot rank results.  Rejected.

|Consequences
a|
* Positive: No separate database server process.  One file to back up.
  Zero network latency for queries.
* Positive: FTS5 provides ranked full-text search sufficient for thousands
  of records.
* Negative: Not suitable if the application ever needs to scale beyond a
  single server or support concurrent write-heavy workloads.
* Negative: FTS5 content-table triggers must be maintained manually.
|===

=== ADR-3 — Leptos Server Functions as the Data Layer

[cols="1,3", options="header"]
|===
|Attribute |Detail

|Status
|Accepted

|Context
|The frontend (WASM) and backend (Axum) must communicate to read and write
bookmark data.  The interface between them must be defined.

|Decision
|Use Leptos server functions (`#[server]` macro) exclusively.
No manually designed REST or GraphQL API.

|Alternatives Considered
a|
* *REST API* — Clean separation, well-understood, easy to test with `curl`.
  Requires manual route definitions, serialisation contracts, and client-side
  fetch wrappers.  Rejected in favour of less boilerplate.
* *GraphQL* — Flexible query language.  Adds a schema and resolver layer with
  no benefit for a single-client application.  Rejected.

|Consequences
a|
* Positive: Zero boilerplate for the data layer.  Adding a new operation
  requires only a Rust function with `#[server]`.
* Positive: Compile-time type safety between UI call sites and server handlers.
* Negative: Server functions are Leptos-specific.  The data layer cannot be
  reused by a non-Leptos client without refactoring.
* Negative: The HTTP endpoints generated by Leptos are not human-readable REST
  endpoints, making external integration harder.
|===

=== ADR-4 — Single Docker Container with Named Volume

[cols="1,3", options="header"]
|===
|Attribute |Detail

|Status
|Accepted

|Context
|The application must be deployable with a single command on a private server.
A containerisation strategy is needed.

|Decision
|Package the application as a single Docker image (multi-stage build).
Use Docker Compose for orchestration.
Mount the SQLite file on a named Docker volume.

|Alternatives Considered
a|
* *Two containers (app + database)* — Only applicable with a client-server
  database (e.g., Postgres).  Not applicable with SQLite file access.
  Rejected.
* *Systemd service without Docker* — Simpler than Docker for a single binary.
  However, Docker provides process isolation, restart policies, and log
  capture that the owner values.  Rejected.

|Consequences
a|
* Positive: `docker compose up -d` starts the application.
  `docker compose down` stops it without touching data.
* Positive: The named volume survives container recreation and image upgrades.
* Positive: `docker logs` gives access to structured application logs.
* Negative: Docker must be installed on the server.
* Negative: WASM compilation and multi-stage builds increase initial build time.
|===

=== ADR-5 — API Key Authentication with Session Cookie

[cols="1,3", options="header"]
|===
|Attribute |Detail

|Status
|Accepted

|Context
|The application is exposed to the internet but must only be accessible by
the owner.  A single-user authentication mechanism is needed.

|Decision
|Protect all routes with a single API key configured via the `API_KEY`
environment variable.  The key is embedded in the bookmarklet URL as a query
parameter.  A successful key validation sets a browser-session cookie so
subsequent navigation does not require the key in the URL.

|Alternatives Considered
a|
* *Username and password form* — Familiar UX but adds a login page interaction.
  Provides no meaningful security benefit over a long API key for a single user.
  Rejected.
* *HTTP Basic Auth* — Browser-native but produces an ugly browser dialog.
  Cannot be embedded in a bookmarklet cleanly.  Rejected.
* *OAuth via GitHub / Google* — No password to manage.  Requires outbound
  network access to an OAuth provider, a registered application, and redirect
  URI management.  Too much complexity for a personal tool.  Rejected.

|Consequences
a|
* Positive: Zero UI friction after the first request.  The bookmarklet
  navigates directly to the add form without a separate login step.
* Positive: No user database, no password hashing, no account management.
* Negative: The API key appears in the bookmarklet URL and is therefore
  visible in browser history and bookmarklet source on the owner's devices.
  Accepted risk for a single-user tool on trusted devices.
* Negative: If the API key is compromised, the only remediation is to
  change the `API_KEY` environment variable and restart the container.
|===

=== ADR-6 — Tag Normalisation at Write Time

[cols="1,3", options="header"]
|===
|Attribute |Detail

|Status
|Accepted

|Context
|Tags are the primary retrieval mechanism.  Inconsistent casing or whitespace
leads to duplicate tags that fragment the collection (e.g., "Rust" and "rust"
appear as separate tags).

|Decision
|Normalise all tags to lowercase and trim surrounding whitespace at write time,
inside the domain layer, before any database write.

|Alternatives Considered
a|
* *Case-sensitive storage with case-insensitive search* — Preserves the author's
  capitalisation but complicates tag autocomplete and filter logic.  Rejected.
* *No normalisation, user responsible* — Places the burden on the owner to be
  consistent.  Incompatible with the simplicity goal.  Rejected.

|Consequences
a|
* Positive: Tag autocomplete and filtering work correctly without any
  case-handling in query code.
* Positive: No duplicate tags caused by capitalisation differences.
* Negative: The owner cannot use mixed-case tags (e.g., "JavaScript" becomes
  "javascript").  Accepted trade-off.
|===
